## Purpose

This document is the first explicit cross-surface matrix for the DRTS Phase 1
authentication / registration / invite / suspend / rotate / revoke /
RBAC-denied / expired / degraded / cancelled flows. It tells the truth about
what is route-materialized in the repo today vs. what is still blocked behind a
named downstream task or external dependency.

The matrix is written per-route-group, not per-product-name. The same
business-domain word ("tenant", "partner", "concierge") may appear under
multiple route groups whenever real authority is split across distinct surfaces.

## Reading Convention

Every row uses one of these classification tags:

- `implemented` — there is real route-level UI evidence in repo today and the
  flow exercises a backend command/query seam (or, for repo-local sign-in, a
  documented local bootstrap).
- `selected-shell stub` — the selected shell expresses the surface but stops
  short of the mutation; the row is parked under an explicit downstream task
  rather than a fake mutation.
- `repo-local bootstrap` — the route is implemented but uses a documented
  local-only bootstrap because the canonical seam is still gated outside repo.
  The downstream owner is named.
- `blocked` — the route does not exist yet; the named downstream task owns the
  materialization. The matrix never silently maps these to other surfaces.
- `delegated` — the flow lives entirely on a different surface by design, and
  the matrix points at that surface as the authority instead of duplicating it.

Evidence is cited as `path:line` so reviewers can verify each row against repo
truth instead of prose summaries.
