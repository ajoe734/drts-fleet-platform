## 0. Purpose

This packet converts the selected prototype mutate actions into an authority-safe
command matrix for Tenant Console plus the overlapping Platform Admin / Ops
Console actions that the execution packet explicitly called out:

- tenant booking create / update / cancel
- invite / suspend / role change
- API key issue / rotate / revoke
- webhook enable / disable / retry / replay if supported
- maintenance / notice / publish / resolve actions that cross into management
  console parity

The goal is simple: downstream UI tasks must only render mutate buttons that map
to a canonical backend command endpoint, or else mark the control as blocked /
hidden until a new backend command exists.

## 1. Reading rules

- `supported` means a canonical endpoint and command contract exist today.
- `supported-overloaded` means the action is supported, but reuses a broader
  command whose semantics the UI must not rename or split on its own.
- `gap` means no authority-safe command endpoint exists yet; the UI must hide
  the action or mark it unavailable.
- `forbidden` means the spec or boundary explicitly says the surface must not
  expose that action in the given realm.
- All POST command rows inherit `Idempotency-Key` discipline from
  `phase1_llm_dev_pack_extracted/phase1_llm_dev_pack/05_engineering_conventions_and_ai_dev_playbook.md`.

## 4. Explicit blockers and hide rules

| ID  | Action                                                               | Why it is blocked or hidden                                                                                                                     |
| --- | -------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| B-1 | tenant webhook retry / replay                                        | No canonical command endpoint exists. UI must not retry from browser or emulate replay by re-posting payloads.                                  |
| B-2 | tenant user resend invite                                            | No backend command exists; keep hidden until `tenant-partner` grows one.                                                                        |
| B-3 | tenant feature flag toggle / tenant override                         | Explicitly platform-only via `/api/admin/flags/*`. Tenant feature visibility is read-only and still has its own read-path gap from `XS-UI-001`. |
| B-4 | tenant booking status edit                                           | Product spec forbids direct status edits; only create / update / cancel commands are allowed from tenant surfaces.                              |
| B-5 | partner-mode tenant-admin actions                                    | Partner mode is restricted to bootstrap, eligibility, booking creation, and minimal result tracking. No admin governance buttons may leak in.   |
| B-6 | tenant-side fare override / dispatch override / no-supply resolution | These are ops or platform authority actions on `/api/orders/*`; tenant detail may display resulting state, but must not expose the buttons.     |

## 5. Downstream guidance

- `TEN-UI-004`, `TEN-UI-005`, `TEN-UI-006`, and `TEN-UI-007` should consume
  this file plus `XS-UI-001` together. If a prototype shows a button absent from
  this matrix, default to hidden until a new backend command is recorded.
- Where this matrix says `supported-overloaded`, keep the UI copy honest. The
  backend command name is the truth; do not split one command into multiple
  client-only endpoint assumptions.
- Where this matrix says `gap`, open or consume a backend task first. Do not
  ship a mock action in the formal tenant-console target.
