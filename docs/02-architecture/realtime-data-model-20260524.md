# Realtime Data Model — 2026-05-24

## 1. Decision

Phase 1 uses a mixed realtime model:

- Web apps use polling and manual refresh.
- Driver App uses polling plus native push for urgent events.
- WebSocket is not part of the Phase 1 baseline.

This avoids adding persistent realtime infrastructure while still supporting dispatch, driver, approval, and operational interruption needs.

## 2. Refresh tiers

| Tier | Interval | Surfaces | Examples |
|---|---:|---|---|
| `fast` | 3–5s | Ops dispatch, call center active call, approval queue | active dispatch board, active phone call, approval escalation |
| `medium` | 15–30s | Ops cases, platform admin health, adapter registry | complaints, incidents, partner adapter health |
| `slow` | 60–120s | tenant lists and settings | bookings list, invoices, webhooks, reports |
| `manual` | user-triggered | reports, audit, long-running artifacts | audit trail, filing package, report jobs |
| `push_interrupt` | event-based | Driver App only | new offer, lost race, reauth, SOS ack |

## 3. Required contract: UiRefreshMetadata

```ts
export interface UiRefreshMetadata {
  generatedAt: string;
  staleAfterMs: number;
  dataFreshness: "fresh" | "stale" | "degraded" | "unknown";
  source: "live" | "cache" | "sandbox" | "static";
  lastSuccessfulRefreshAt?: string | null;
  nextSuggestedRefreshAt?: string | null;
  refreshReason?: string | null;
}
```

## 5. API behavior

Every list/detail endpoint that participates in polling must either:

1. embed `refresh: UiRefreshMetadata`, or
2. expose a paired `/health` or `/summary` endpoint that returns `UiHealthEnvelope`.

Minimum required endpoints:

```http
GET /api/ops/dispatch/summary
GET /api/ops/callcenter/active-sessions
GET /api/ops/approval-requests/summary
GET /api/admin/health/ui
GET /api/admin/tenants/{tenantId}/readiness
GET /api/tenant/integration-readiness
GET /api/driver/workspace/summary
GET /api/driver/platform-presence/summary
```

## 6. UI rules

- UI must display stale or degraded data when `dataFreshness != fresh`.
- UI must not silently hide stale data when last known data is useful.
- UI must not claim live data when `source` is `static` or `sandbox`.
- Manual refresh buttons must show loading and last successful refresh time.
- Push events in Driver App must be confirmed by pull API before authoritative state changes.

## 7. Acceptance criteria

- `UiRefreshMetadata` and `UiHealthEnvelope` are exported from contracts.
- Each app consumes at least one refresh metadata response.
- Ops dispatch and driver workspace show stale/degraded state in tests.
- No WebSocket dependency is introduced for Phase 1.
