# Notification Inbox Contract — 2026-05-24

## 1. Decision

Phase 1 uses backend-owned per-user notification inboxes. Driver App additionally uses native push for urgent events.

No app owns notification truth locally.

## 2. Required contract: UserNotificationRecord

```ts
export type UserNotificationSeverity = "info" | "warning" | "critical";
export type UserNotificationStatus = "unread" | "read" | "archived";
export type UserNotificationChannel =
  | "ops_notice"
  | "tenant_sla"
  | "driver_task"
  | "tenant_approval"
  | "platform_admin"
  | "partner_booking";

export interface UserNotificationRecord {
  notificationId: string;
  recipientUserId: string;
  tenantId: string | null;
  actorRealm: "platform" | "ops" | "tenant" | "driver" | "partner";
  channel: UserNotificationChannel;
  severity: UserNotificationSeverity;
  title: string;
  message: string;
  status: UserNotificationStatus;
  eventType: string;
  resourceLinks: CrossAppResourceLink[];
  action?: ResourceActionDescriptor | null;
  createdAt: string;
  readAt: string | null;
  expiresAt?: string | null;
  metadata?: Record<string, unknown>;
}
```

## 3. Notification events

### Ops Console

- dispatch no eligible supply
- approval request pending / breached
- CTI recording pending / attached
- complaint SLA breached
- incident escalation
- adapter sync failure
- reconciliation assigned

### Platform Admin

- tenant rollout blocked
- partner credential expiring
- adapter unhealthy
- pricing publish requires review
- production rollout hold
- legal hold / deletion exception

### Tenant Console

- booking created / updated / cancelled
- approval requested / approved / rejected / escalated
- quota warning / exceeded
- webhook delivery failed
- report ready
- invoice issued

### Driver App

- new task offer
- lost race
- cancelled by platform
- reauth required
- incident hold
- proof required
- SOS acknowledged

## 4. API surface

```http
GET  /api/notifications
POST /api/notifications/read
POST /api/notifications/archive
GET  /api/notifications/summary
```

Driver-specific urgent push is not a source of truth. Push payloads must include an event id and resource id, and Driver App must call API to confirm.

## 5. UI behavior

- Notification center must show unread count by severity.
- Notifications must link to resources using `CrossAppResourceLink`.
- Notifications may embed one backend-provided `ResourceActionDescriptor` for quick action.
- UI must not create notifications locally except transient toasts; transient toasts are not persisted notification records.

## 6. Acceptance criteria

- `UserNotificationRecord` exported from contracts.
- Notification list and mark-read APIs exist or current endpoints are amended to match this shape.
- Tenant approval request, driver task, and platform adapter unhealthy events each create inbox records in tests.
- Driver push confirmation path is documented and does not mutate authoritative state without pull confirmation.
