# UI Authority Actions Contract — 2026-05-24

## 1. Decision

UI actions must be backend-authorized. Frontend may render actions but cannot decide canonical permission or state transitions by local role checks alone.

## 4. Risk rules

| Risk | UI requirement |
|---|---|
| low | inline or direct action permitted; receipt still required |
| medium | confirmation modal required |
| high | confirmation modal + reason required; receipt must show audit reference |

## 5. Resource action endpoints

Minimum endpoints:

```http
GET /api/admin/tenants/{tenantId}/actions
GET /api/admin/partners/{entrySlug}/actions
GET /api/ops/orders/{orderId}/actions
GET /api/ops/complaints/{caseNo}/actions
GET /api/tenant/bookings/{bookingId}/actions
GET /api/tenant/approval-requests/{approvalRequestId}/actions
GET /api/driver/tasks/{taskId}/actions
GET /api/driver/platforms/{platformCode}/actions
```

Endpoints may be consolidated under a generic action service if typed wrappers exist.

## 6. CTA rendering rule

- `available`: button/menu item enabled.
- `disabled`: show disabled with reason when helpful.
- `hidden`: do not render, but keep testable in action descriptor response.

Frontend hard-coded role checks may narrow display for user experience, but cannot widen action availability.

## 7. Acceptance criteria

- `ResourceActionDescriptor` and `ActionReceipt` exported.
- At least one write flow in each app returns and displays `ActionReceipt`.
- At least one high-risk action requires reason and returns audit reference.
- Tests verify UI cannot enable action when descriptor says disabled.

## 8. Repo-local implementation decision — 2026-05-25 (ADR `PACK-LAND-202605`)

The 8 endpoints listed in §5 are **not implemented as standalone routes**. Pack §5 explicitly allows consolidation. The repo adopts the consolidated pattern:

1. Detail and list responses for each affected `resourceType` embed `availableActions: ResourceActionDescriptor[]` directly on the record envelope.
2. The reference implementation lives in [apps/api/src/modules/owned-mobility/owned-mobility.service.ts](../../apps/api/src/modules/owned-mobility/owned-mobility.service.ts) (BookingRecord / OwnedOrder availableActions builder). Other modules adopting Pack contracts must follow this pattern.
3. When more than two modules need shared descriptors, the builder factors out to a `apps/api/src/common/resource-actions/` helper. Until then, per-module builders are acceptable.
4. UI clients MUST consume the embedded `availableActions` array from the same response that returned the resource. UI MUST NOT call a separate `*/actions` endpoint, because none is implemented.
5. Audit + tests for the consolidated pattern remain identical to pack acceptance: a `availability="disabled"` descriptor must reach the UI, and the UI test must verify the corresponding mutation is not invocable.

This decision applies to all 8 endpoints in §5 unless a future amendment reverses it. Wave `phase1-ui-implementation-wave-202605` Layer 1 BE tasks adopt this pattern; Layer 3 FE tasks consume embedded `availableActions` accordingly.
