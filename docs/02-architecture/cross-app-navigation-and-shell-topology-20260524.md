# Cross-App Navigation and Shell Topology — 2026-05-24

## 1. Decision

Phase 1 keeps independent deployments:

```text
platform-admin-web
ops-console-web
tenant-console-web
driver-app
partner-booking-web
```

There is no unified shell, no module federation, and no multi-app SPA in Phase 1.

## 2. Rationale

- Each app has different users, permissions, runtime expectations, and risk profile.
- Platform Admin is a high-sensitivity control plane.
- Ops Console is an operational workbench.
- Tenant Console is an external tenant admin surface.
- Driver App is a mobile app with device, location, push, and offline considerations.
- Partner Booking is constrained partner entry, not tenant admin.

## 3. Required contract: CrossAppResourceLink

```ts
export type CrossAppSurface =
  | "platform_admin"
  | "ops_console"
  | "tenant_console"
  | "driver_app"
  | "partner_booking";

export interface CrossAppResourceLink {
  label: string;
  surface: CrossAppSurface;
  resourceType:
    | "tenant"
    | "partner_entry"
    | "booking"
    | "order"
    | "driver"
    | "vehicle"
    | "complaint_case"
    | "incident"
    | "invoice"
    | "report"
    | "audit_event"
    | "approval_request"
    | "platform_task"
    | "reconciliation_issue";
  resourceId: string;
  href: string;
  access: "allowed" | "forbidden" | "unknown";
  reasonCode?: string | null;
  opensInNewContext?: boolean;
}
```

## 4. Link generation authority

Backend must produce `CrossAppResourceLink[]` where resource-level permission is complex.

Frontend may construct static app navigation links only for same-surface route navigation. Cross-surface links attached to data records should come from API or typed helper functions backed by shared route registry.

## 5. Deep link patterns

| Target | Pattern |
|---|---|
| Platform tenant detail | `/tenants/{tenantId}` |
| Platform partner entry | `/partners/{entrySlug}` |
| Ops order detail | `/dispatch/orders/{orderId}` |
| Ops complaint case | `/complaints/{caseNo}` |
| Tenant booking detail | `/bookings/{bookingId}` |
| Tenant approval request | `/approval-requests/{approvalRequestId}` |
| Driver task | driver deep link `drtsdriver://tasks/{taskId}` |
| Partner booking entry | `/{entrySlug}` or host-resolved entry |

## 6. Session continuity

- Apps may share identity provider and backend session family, but must not share frontend shell state.
- Cross-app navigation must handle forbidden links gracefully using `access` and `reasonCode`.
- Driver App deep links must not expose management-only resources.

## 7. Acceptance criteria

- `CrossAppResourceLink` exported from contracts.
- Ops order detail, Platform Admin partner detail, Tenant booking detail, and Driver task detail each expose at least one valid cross-app link where appropriate.
- Forbidden cross-app link is displayed disabled or hidden according to product rules.
- No app imports another app's runtime components to create a unified shell.
