# Amendment 2026-05-24 — Platform Admin / Ops Console / Tenant Console Product Specification

**Amends:** `docs/01-product/platform-admin-ops-tenant-console-product-spec-20260508.md`  
**Applies to:** `apps/platform-admin-web`, `apps/ops-console-web`, `apps/tenant-console-web`  
**Status:** binding Phase 1 amendment  
**Reason:** system-design answers define cross-cutting UI behavior, backend action authority, notification inbox, search, empty state, health, rollout and integration readiness requirements that must be added to the product spec.

## 1. Product-spec supersession

This amendment does not replace the 2026-05-08 product spec. It adds binding cross-cutting product requirements. If any page-level implementation or visual-design packet conflicts with this amendment, this amendment wins.

## 2. Additional hard rules

Add the following to the product principles section.

### 2.1 Backend-owned UI behavior

Backend authority must determine:

- whether data is fresh, stale, degraded, sandbox, or static;
- whether a resource can be mutated;
- which actions are available to the current actor;
- whether a resource is not ready, empty, error, restricted, or externally gated;
- whether the user sees a rollout warning, integration warning, or action block;
- which audit/action receipt is returned after a write command.

Frontend may hide, group, style, or prioritize backend-provided actions, but it may not invent or override permission semantics.

### 2.2 No Phase 1 unified shell

`platform-admin-web`, `ops-console-web`, and `tenant-console-web` remain independent apps. They must not be combined into a unified app shell or module-federated management application in Phase 1.

Cross-app movement uses `CrossAppResourceLink` and deep links, not embedded sub-apps.

### 2.3 Mixed realtime model

No WebSocket baseline in Phase 1. Web apps use polling tiers and manual refresh. Urgent native push is reserved for Driver App and operational interrupts only.

### 2.4 Action receipt requirement

Every write command must return an `ActionReceipt` or an existing response shape that includes an equivalent `actionId`, `requestId`, `auditId`, `actor`, `timestamp`, and affected resource summary.

This applies to:

- tenant rollout stage changes;
- partner credential issue / revoke;
- pricing publish;
- feature flag changes;
- vehicle dispatchability changes;
- driver platform offlining;
- call center booking creation;
- complaint reopen / close;
- approval approve / reject / escalate;
- webhook rotation;
- API key issue / revoke;
- billing/reconciliation actions;
- legal hold / deletion exception actions.

## 3. Required shared contracts for all management surfaces

The three management apps must consume the following shared contracts.

| Contract | Management-console use |
|---|---|
| `UiRefreshMetadata` | every list/detail response that is refreshed or may be stale |
| `UiHealthEnvelope` | health bars, degraded banners, integration readiness cards |
| `ResourceActionDescriptor` | all CTAs and command menus |
| `EmptyStateEnvelope` | empty states, blocked setup, no eligible data, external-gated state |
| `ActionReceipt` | all mutation success and error-confirmed receipts |
| `UserNotificationRecord` | per-user backend inbox |
| `CrossAppResourceLink` | safe cross-app navigation |
| `SearchResultRecord` | scoped search and local resource search |
| `TenantIntegrationReadinessSummary` | tenant-console integration dashboard and platform-admin tenant readiness |
| `TenantRolloutStateMachineRecord` | platform-admin rollout governance |

## 4. Surface-specific amendments

### 4.1 Platform Admin

Add product requirements:

1. Tenant rollout state must be rendered from `TenantRolloutStateMachineRecord`; UI cannot infer next stages from status strings.
2. Tenant readiness panels must use `TenantIntegrationReadinessSummary` and must distinguish sandbox, pilot, production, external-gated, rollback hold, and missing-evidence states.
3. Partner credentials must expose backend-provided `availableActions`; credential UI cannot display rotate/revoke/issue based only on role.
4. Adapter registry and switchboard pages must use `UiHealthEnvelope` for provider health and `UiRefreshMetadata` for stale/degraded data.
5. Pricing publish and fee-plan changes must return `ActionReceipt` and display an audit reference to the user.
6. Legal hold and deletion exception actions must require reason capture and return an audit receipt.

### 4.2 Ops Console

Add product requirements:

1. Dispatch, call center, approval queue, complaint, incident, maintenance, and revenue pages must use fixed polling tiers; no Phase 1 WebSocket dependency.
2. Operator CTAs must use `ResourceActionDescriptor` and must show unavailable actions with reason where operationally helpful.
3. Driver offlining per platform must be represented through backend action descriptors and `DriverOpsInstruction` when the driver must receive an instruction.
4. Incident or complaint driven matching suppression must be represented by `DriverMatchingSuppression`; Ops UI may display a suppression but not invent one locally.
5. Call center recording and filing states must render health/freshness and not over-claim live provider evidence when the provider is sandbox or static.
6. Revenue mismatch details must include `CrossAppResourceLink` to related orders, driver statement, partner entry, invoice, and reconciliation issue where authorized.

### 4.3 Tenant Console

Add product requirements:

1. `tenant-console-web` is the repo-local canonical tenant admin console. `tenant-commute-hub` remains transition/live owner until a cutover record says otherwise.
2. Tenant Console must not own backend truth for bookings, cost centers, approval rules, quotas, users, API keys, webhooks, billing, reports, or audit.
3. Integration readiness must be displayed through `TenantIntegrationReadinessSummary`.
4. Cost center, quota, approval, and booking action buttons must use `ResourceActionDescriptor` or route-specific command responses with action descriptors.
5. Empty states for reports, invoices, API keys, webhooks, cost centers, and rules must use `EmptyStateEnvelope`.
6. Tenant user notification preferences and inbox must use `UserNotificationRecord` and backend subscriptions; no local-only notification truth.
