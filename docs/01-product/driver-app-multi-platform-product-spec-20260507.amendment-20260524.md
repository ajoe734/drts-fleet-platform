# Amendment 2026-05-24 — Driver App Multi-Platform Product Specification

**Amends:** `docs/01-product/driver-app-multi-platform-product-spec-20260507.md`  
**Applies to:** `apps/driver-app`, `platform-presence`, `platform-earnings`, `forwarder`, `owned-mobility`, `driver-profile`, `shift-attendance`  
**Status:** binding Phase 1 amendment

## 1. Product-spec supersession

This amendment strengthens the existing Driver App product requirement: the Driver App is a multi-platform driver workbench, not a single-platform owned-dispatch app.

The existing product spec already states that the driver should use one app to receive, understand, and operate work from DRTS owned dispatch and multiple external ride-hailing / taxi platforms. This amendment defines the additional UI/contract behavior required to make that product claim enforceable.

## 2. Required new driver contracts

Driver App and related Ops workflows must consume:

| Contract | Driver / Ops use |
|---|---|
| `DriverOpsInstruction` | backend-authored instruction sent to driver UI for reauth, platform offline, incident hold, manual fallback, proof required, lost race, sync failed, etc. |
| `DriverMatchingSuppression` | backend-authored reason a driver is temporarily excluded from matching or from a specific platform/task class |
| `UiRefreshMetadata` | task inbox, trip state, platform presence, earnings freshness |
| `UiHealthEnvelope` | platform account / adapter / device health summary |
| `ResourceActionDescriptor` | accept, reject, start, arrive, complete, upload proof, reauth, go offline actions |
| `ActionReceipt` | confirmation after accept/reject/platform presence change/proof upload/SOS |
| `UserNotificationRecord` | driver inbox for non-urgent notifications; urgent events still use native push |
| `CrossAppResourceLink` | driver-safe links where applicable; deep links must not expose management-only resources |

## 3. Driver App flow amendments

### 3.1 Platform binding and presence

Driver platform presence must be represented as a backend-owned state. UI can ask to change state, but cannot directly mutate platform availability.

Required states:

```text
not_bound
bound_offline
bound_online
reauth_required
suspended
incident_hold
degraded
```

Every platform card must show:

- platform code and label;
- online/offline state;
- token/credential health if applicable;
- next required instruction if any;
- available actions from `ResourceActionDescriptor[]`.

### 3.2 Driver task inbox

A task row must distinguish:

- owned DRTS task;
- forwarded external-platform task;
- route locked vs local route;
- accept-pending external race;
- lost race;
- platform cancellation;
- sync failed;
- manual fallback;
- proof pending.

UI must not allow owned-trip lifecycle actions on forwarded tasks unless backend explicitly provides the action.

### 3.3 Driver trip detail

Trip detail must render a `DriverOpsInstruction` when backend requires driver attention.

Examples:

```text
reauth_required
platform_sync_failed
lost_race
cancelled_by_platform
proof_required
manual_fallback_pending
incident_hold
matching_suppressed
```

### 3.4 Offline and replay semantics

Phase 1 allows limited offline resilience but not offline authority.

Allowed offline behavior:

- display last known task list with stale marker;
- queue non-mutating telemetry locally;
- queue proof upload draft until connectivity returns;
- show safe fallback instructions.

Disallowed offline behavior:

- accept a new external-platform offer while offline;
- start or complete a forwarded task without backend confirmation;
- modify route/fare/settlement authority locally;
- override lost-race or platform cancellation state.

### 3.5 Matching suppression

`DriverMatchingSuppression` must be shown in workspace, platform presence, and task-matching contexts when relevant.

Suppression can be caused by:

- incident hold;
- expired license / certificate;
- vehicle ineligible;
- platform credential invalid;
- platform-specific suspension;
- shift offline;
- manual ops hold;
- legal/compliance hold.

UI can display suppression and direct driver to next action, but cannot clear it locally.

## 4. Driver urgent push vs polling

Driver App must use native push only for urgent interrupts:

- new task offer;
- task cancelled by platform;
- lost race;
- reauth required while online;
- SOS acknowledgement;
- incident hold;
- proof required before completion;
- high-priority Ops instruction.

All pushed events must be confirmed by pull API before UI changes authoritative state.

## 5. Required product-spec insertion

Implementation team should amend the target spec with a new chapter:

```text
## 12. 2026-05-24 Driver Ops Instruction, Matching Suppression, and Offline Contract Addendum
```

The chapter must include:

- platform binding states;
- driver task authority boundaries;
- forwarded task restrictions;
- offline/replay rules;
- native push confirmation rule;
- `DriverOpsInstruction` use cases;
- `DriverMatchingSuppression` use cases;
- action descriptor / action receipt requirements.

## 6. Acceptance criteria

- Driver App compiles with all new contracts.
- Task inbox and trip detail can render `DriverOpsInstruction`.
- Platform presence can render `DriverMatchingSuppression` and health metadata.
- Accept/reject/complete/proof/SOS paths return or surface an `ActionReceipt`.
- Offline task view displays stale data through `UiRefreshMetadata`.
- No forwarded task action is enabled without backend `ResourceActionDescriptor`.
